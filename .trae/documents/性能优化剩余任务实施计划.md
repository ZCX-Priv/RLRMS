## 关键决策

### 决策1：DishesView 虚拟滚动（任务6）- 跳过

**探索发现**：`src/admin/views/DishesView.vue` 第 438-510 行使用 `vuedraggable` 实现菜品拖拽排序，这是核心交互功能。`vue-virtual-scroller` 的 `RecycleScroller`/`DynamicScroller` 与 `vuedraggable` 的集成需要复杂的自定义实现，且：

1. 菜品数量通常为几十到几百量级，非万级列表，性能瓶颈不显著
2. 拖拽排序是管理员核心功能，破坏它代价过高
3. DashboardView 的订单列表已实现虚拟滚动（任务6 主要受益场景）

**结论**：跳过 DishesView 的虚拟滚动改造，保持拖拽功能完整。任务6 在 DashboardView 侧已完成，整体目标达成。

### 决策2：vite.config.ts esbuild 配置（任务18）- 使用 pure 保留 warn/error

**探索发现**：原 `removeConsolePlugin`（第 9-25 行）仅移除 `console.log/info/debug`，**保留 `console.warn` 和 `console.error`**，这对生产环境排查问题至关重要。

若直接使用 `esbuild: { drop: ['console'] }`，会移除**所有** console 调用，包括 warn/error，导致生产环境丢失重要警告日志，属于行为回归。

**结论**：使用 `esbuild.pure: ['console.log', 'console.info', 'console.debug']` 精确匹配原插件行为，保留 warn/error。同时添加 `drop: ['debugger']` 移除 debugger 语句。

### 步骤1：任务12 - HomeView.vue 动画性能优化

**文件**：`c:\Users\赵晨旭\Desktop\红灯笼食府管理系统\src\client\views\HomeView.vue`

**问题**：第 712-766 行的 `cartSlideIn`/`cartSlideOut` 动画使用 `max-height`（从 0 到 40vh），`max-height` 会触发重排（reflow），性能较差。

**修改内容**：
- 第 712-725 行 `@-webkit-keyframes cartSlideIn`：移除 `max-height`，保留 `opacity` 和 `transform: translateY()`
- 第 727-738 行 `@keyframes cartSlideIn`：同上
- 第 740-753 行 `@-webkit-keyframes cartSlideOut`：移除 `max-height`，保留 `opacity` 和 `transform: translateY()`
- 第 755-766 行 `@keyframes cartSlideOut`：同上

**修改后示例**（cartSlideIn）：
```css
@keyframes cartSlideIn {
  0% {
    opacity: 0;
    transform: translateY(20px);
  }
  100% {
    opacity: 1;
    transform: translateY(0);
  }
}
```

**原理**：`transform` 和 `opacity` 是合成层（compositor-only）属性，不触发重排，动画更流畅。

**注意**：`.cart-expanded` 容器本身需有明确的高度或由内容撑开，移除 `max-height` 动画后，容器高度由内容决定。需确认第 285-286 行的 `<Transition name="cart-expand">` + `<div v-if="showCart" class="cart-expanded">` 结构在无 max-height 时仍正常显示。

### 步骤2：任务15 - DashboardView.vue AudioContext 单例

**文件**：`c:\Users\赵晨旭\Desktop\红灯笼食府管理系统\src\admin\views\DashboardView.vue`

**问题**：第 252-282 行 `playNotificationSound` 函数每次调用都 `new AudioContext()`，浏览器对 AudioContext 实例数有限制（通常 6 个），频繁创建会导致报错或资源泄漏。

**修改内容**：
1. 在 `<script setup>` 顶部（第 240 行 `formatDate` 函数前或后）添加模块级单例变量：
```typescript
// AudioContext 单例：避免频繁创建导致资源耗尽
let audioContextInstance: AudioContext | null = null

/**
 * 获取共享的 AudioContext 实例
 * 浏览器对 AudioContext 实例数有限制，复用单例避免资源耗尽
 */
function getAudioContext(): AudioContext {
  if (!audioContextInstance) {
    const Ctor = window.AudioContext || (window as unknown as { webkitAudioContext: typeof AudioContext }).webkitAudioContext
    audioContextInstance = new Ctor()
  }
  return audioContextInstance
}
```

2. 修改 `playNotificationSound`（第 252-282 行），将 `new (window.AudioContext || ...)` 替换为 `getAudioContext()`：
```typescript
function playNotificationSound() {
  try {
    const audioContext = getAudioContext()
    // 其余 oscillator 逻辑保持不变
    ...
  } catch (e) {
    console.log('Audio notification not supported')
  }
}
```

**注意**：保留原有 oscillator 逻辑（两个连续的蜂鸣音）不变，仅替换 AudioContext 的获取方式。

### 步骤3：任务17 - OrderConfirmView.vue handleNameInput debounce

**文件**：`c:\Users\赵晨旭\Desktop\红灯笼食府管理系统\src\client\views\OrderConfirmView.vue`

**问题**：第 106-112 行 `handleNameInput` 每次按键都触发 `validateName` + `setItem`（IndexedDB 写入），高频输入时造成不必要的计算和 IO。

**探索发现**：项目中无现成 debounce 工具函数（`src/utils/` 仅有 `storage.ts`，`src/shared/composables/` 仅有 `useOrderPolling`、`useDragReorder`、`useStaggeredAnimation`）。

**修改内容**：
1. 在 `<script setup>` 中（第 31 行 `nameError` ref 附近）添加 debounce 定时器变量和工具函数：
```typescript
// 姓名输入校验的 debounce 定时器
let nameValidateTimer: ReturnType<typeof setTimeout> | null = null
const NAME_INPUT_DEBOUNCE_MS = 300

/**
 * 带防抖的姓名校验与持久化
 * 避免高频输入时重复执行正则匹配和 IndexedDB 写入
 */
function debouncedValidateAndPersist(value: string) {
  if (nameValidateTimer) {
    clearTimeout(nameValidateTimer)
  }
  nameValidateTimer = setTimeout(() => {
    validateName(value)
    setItem('contact_name', value)
    nameValidateTimer = null
  }, NAME_INPUT_DEBOUNCE_MS)
}
```

2. 修改 `handleNameInput`（第 106-112 行），仅同步更新 `contactName.value`，校验和持久化走 debounce：
```typescript
function handleNameInput(event: Event) {
  const value = (event.target as HTMLInputElement).value
  // 同步更新输入值，保证输入框响应即时
  contactName.value = value
  // 防抖处理校验与持久化，减少高频计算和 IO
  debouncedValidateAndPersist(value)
}
```

**注意**：`contactName.value` 同步更新保证输入框不卡顿；`validateName` 和 `setItem` 延迟 300ms 执行。`canSubmit` computed 依赖 `contactName.value` 和 `nameError.value`，由于 `nameError` 延迟更新，提交按钮可能在 300ms 内显示旧状态——这是可接受的权衡（用户停止输入后才触发校验是常见 UX）。

### 步骤5：任务19 - DishesView.vue specs 输入处理提取

**文件**：`c:\Users\赵晨旭\Desktop\红灯笼食府管理系统\src\admin\views\DishesView.vue`

**问题**：第 618-623 行的 specs 输入框使用内联复杂表达式：
```html
<input
  v-model="formData.specs"
  type="text"
  placeholder="大份,中份,小份"
  @input="formData.specs = ($event.target as HTMLInputElement).value.split(',').map(s => s.trim()).filter(Boolean)"
/>
```
存在两个问题：
1. 模板内联复杂逻辑，可读性差，违反关注点分离
2. `v-model="formData.specs"` 绑定数组到 input，与 `@input` 冲突（v-model 会用 input.value 覆盖，但 specs 是数组，显示异常）

**修改内容**：
1. 在 `<script setup>` 中（`formData` 定义之后，约第 34 行附近）添加 `handleSpecsInput` 方法：
```typescript
/**
 * 处理规格输入：将逗号分隔的字符串解析为数组
 * 提取自模板内联表达式，提升可读性
 */
function handleSpecsInput(event: Event) {
  const raw = (event.target as HTMLInputElement).value
  formData.value.specs = raw
    .split(',')
    .map(s => s.trim())
    .filter(Boolean)
}
```

2. 修改模板第 616-624 行，移除 `v-model`（避免数组绑定冲突），改用 `:value` 显示 + `@input` 处理：
```html
<div class="form-group full-width">
  <label>规格 (用逗号分隔)</label>
  <input
    :value="formData.specs.join(', ')"
    type="text"
    placeholder="大份,中份,小份"
    @input="handleSpecsInput"
  />
</div>
```

**注意**：`:value="formData.specs.join(', ')"` 将数组以逗号空格连接显示；用户输入时 `handleSpecsInput` 解析回数组。这样避免了 v-model 与数组类型的冲突，且显示一致。

## 风险与回滚

- **任务12 风险**：移除 `max-height` 动画后，若 `.cart-expanded` 容器无明确高度，展开时可能无过渡效果。回滚方案：恢复 max-height 动画。

- **任务19 风险**：`:value` + `@input` 替代 `v-model` 后，若光标位置跳动，需改用 `v-model` + `watch` 方案。当前方案假设 input 的 value 由 specs.join 驱动无光标问题。
